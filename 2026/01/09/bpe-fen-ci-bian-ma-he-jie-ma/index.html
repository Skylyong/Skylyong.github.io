在这里，我们介绍 BPE Tokenizer的训练方法，接下来我们将介绍如何利用训练阶段得到的词表（Vocabulary）字典，以及 Merge记录来对新的文本进行编码和解码。

BPE Tokenizer Encoding

什么是 BPE Tokenizer Encoding

BPE Tokenizer Encoding 是指利用 训练时得到的词表（Vocabulary）和 Merge 将文本编码为 整数ID的过程。

BPE Tokenizer Encoding 的核心步骤

BPE Tokenizer Encoding 包含三个核心步骤，分别为：

  1. 预分词 (Pre-tokenize)：
  • 首先将输入文本分割成“预分词单元”（pre-tokens）。
  • 将每个预分词单元转换为 UTF-8 字节序列。
  • 关键点：每个预分词单元是独立处理的，不会跨越预分词边界进行合并。
  1. 应用合并规则 (Apply the merges)：
  • 使用训练阶段生成的合并规则列表（merges list）。
  • 顺序：严格按照规则生成的顺序依次应用。
  • 过程：在每个预分词单元内部，查找并合并符合规则的字节对，直到没有更多的规则适用或者没有可合并的字节对为止。
  1. 映射为整数 ID (Map to IDs)：
  • 将合并处理后最终形成的词汇单元（vocabulary elements），根据词表映射为对应的整数（Token IDs）。

BPE Tokenizer Encoding 中需要特殊处理的内容

BPE Tokenizer Encoding 中有两个内容需要特别处理：

  1. 特殊 Token (Special Tokens)：
  • 编码器必须能够正确识别和处理用户定义的特殊 Token（如 [CLS], [SEP] 等），防止它们被错误分解。
  1. 内存管理与分块处理 (Memory Considerations)：
  • 场景：处理无法一次性装入内存的大型文本文件或数据流。
  • 方法：将数据切分成可管理的“块”（chunks）依次处理，以保持恒定的内存复杂度。
  • 约束：必须确保 Token 不会跨越块的边界（即不能把一个完整的 Token 切分在两个块中），以保证分块处理的结果与一次性全量处理的结果完全一致。

对 “应用合并规则”的详细解释

合并规则的核心逻辑为按优先级”查表”，想象手中有一份合并排行榜 （即训练阶段得到的 merges 列表）。这个排行榜是有严格顺序的：排名越靠前，优先级越高。
对于每一个预分词单元 （Pre-token），我们需要做的是：

  1. 把它看作一串基础的字节序列。
  2. 拿出 “合并排行榜”，从第一名开始往下查。
  3. 如果当前序列中出现了排行榜上的“字节对”，就立即把它们合并成一个新的单元。
  4. 重复这个过程，直到排行榜查完，或者序列中再也找不到可以合并的组合为止。

BPE Tokenizer Encoding 的实例演示

假设我们的预分词单元是 [‘the’, ‘ cat’, ‘ ate’]。
假设我们的**“合并排行榜（Merges）”**（按顺序）如下：
(b’t’, b’h’) 👉 (碰到 t 和 h 就合并)
(b’ ‘, b’c’) 👉 (碰到 空格 和 c 就合并)
(b’ ‘, b’a’) 👉 (碰到 空格 和 a 就合并)
(b’th’, b’e’) 👉 (碰到 th 和 e 就合并)
(b’ a’, b’t’) 👉 (碰到 “ a” 和 t 就合并)

  2. 确定性（Deterministic）：只要词表和合并规则列表确定，对于同样的输入文本，永远会生成同样的 Token ID 序列。这保证了模型训练和推理时的一致性。

代码实现

BPE Tokenizer Decoding

BPE Tokenizer Decoding 指的是将整数ID解码为原始的文本，相对于 Encoding， Decoding过程比较简单。主要包含两方面的内容：

标准解码流程

  • 查表与拼接： 查找每个 Token ID 在词表中对应的字节序列，并将它们按顺序拼接在一起。
  • 转换字符串： 将拼接后的完整字节序列解码为 Unicode 字符串。

异常处理（无效编码）

  • 问 题：由于输入的 Token ID 序列可能是任意组合，拼接后的字节序列不一定能构成合法的 Unicode 字符。
  • 解决方案：如果遇到无法识别的畸形字节，不能报错中断，而是要将其替换为标准的 Unicode 替换字符 (U+FFFD)。
  • 代码实现：在执行 bytes.decode 时，设置参数 errors=’replace’，系统会自动将非法字节转换为替换标记。
