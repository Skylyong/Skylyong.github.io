        4. [!summary]

        • Why we need tokenizer？
        • 为什么需要基于字节的分词器？
        • 彻底解决”未登录词”问题

        1. The motivation - Why we need tokenizer

        由于计算机只能够对数字进行计算，而没法直接对字符做计算。因此，我们需要一种操作能够将字符串编码为数字；同时将计算机输出的数字解码为人类阅读友好的字符串。
        上述中的“操作”即为 Tokenizers。

        根据编码基本单元的不同，可以划分出多种不同的编码方式，如基于词级、字符级和基于字节的编码方式。

        其中，基于字节的Tokenizer（Byte-based Tokenizer），或者更准确地说是字节级（Byte-Level）子词分词器，是现代大语言模型（如 GPT-2, GPT-3, GPT-4, Llama, Claude 等）中广泛使用的核心技术。

        它的核心理念是：将文本的基础构建块从“字符（Character）”下沉到“字节（Byte）”，从而彻底解决“未登录词（OOV, Out-Of-Vocabulary）”问题，实现对所有语言和符号的通用支持。

        以下是对基于字节的Tokenizer的详细解释，包括其背景、原理、工作流程及优缺点。

        2. 为什么要用基于字节的Tokenizer？（背景）

        在字节级分词出现之前，NLP模型主要使用以下两种方式：

        1. 词级（Word-level）： 把“apple”看作一个整体。
          • 缺点： 词汇表太大（几十万个），且无法处理从未见过的词（OOV问题），只能用 <UNK> 代替。
        2. 字符级（Character-level）： 把“apple”拆成 a-p-p-l-e。
          • 缺点： 序列太长，且单个字符缺乏语义。

        后来出现了 BPE（Byte-Pair Encoding） 等子词算法，试图平衡两者。但在传统的BPE中，基础单元仍然是 Unicode 字符。如果遇到生僻字（如罕见的emoji或古汉字），如果它不在词表中，仍然会变成 <UNK>。

        基于字节的Tokenizer 的出现就是为了解决这个问题：计算机里的所有信息本质上都是字节。只要我们基于字节进行处理，就没有表达不了的内容。

        3. 核心原理：UTF-8 与 字节

        理解这个技术的前提是理解 UTF-8 编码。UTF-8 是一种变长编码：

        • 英文/数字（ASCII）： 占用 1个字节。
        • 中文/日文/韩文等： 通常占用 3个字节。
        • Emoji 表情： 通常占用 4个字节。

        传统 BPE vs. 字节级 BPE (BBPE)：

        • 传统 BPE： 基础词表是 Unicode 字符集（比如几万个汉字、字母）。如果遇到一个没见过的 Unicode 字符，就崩溃了。
        • 字节级 BPE： 初始词表由所有可能的单字节值组成，即 0x00 到 0xFF，共 256 个条目。 必要时，还可以加入特殊标记（如 <pad>、<unk> 等）。
          • 无论什么文字，首先都转成字节流。
          • 然后在字节流上运行 BPE 合并算法。

        [!info] Capture 基础词表

        • 基础词表是 所有可能的单字节值，共256个条目，即一个字节。
        • 为什么用 256 个条目能够表示所有的 字符集合呢？ 因为在UTF-8编码中，还会使用多个字节来表示字符。比如用2字节的话，则可以表示 256x256个字符，大大扩展了其表示空间。

        4. 工作流程详解

        以 GPT-2 引入的 Byte-Level BPE 为例，处理流程如下：

        第一步：文本转字节（Text to Bytes）

        输入文本：”Hi 世界”

        在 UTF-8 编码下，这串文本会被转换成字节序列：

        • H -> 0x48
        • i -> 0x69
        • 空格 -> 0x20
        • 世 -> 0xE4, 0xB8, 0x96 (3个字节)
        • 界 -> 0xE7, 0x95, 0x8C (3个字节)

        原始字节序列： [48, 69, 20, E4, B8, 96, E7, 95, 8C]

        第二步：初始化词表（Vocabulary Initialization）

        基础词表不再是几万个字，而是包含所有可能的 256 个字节值。这意味着任何文件、任何文本都可以被初始表示，绝不会出现 <UNK>。

        第三步：应用 BPE 合并（Merging）

        模型在训练阶段会统计字节组合的频率，并进行合并。

        1. 高频合并： 英文单词通常由 ASCII 字节组成，合并很快。例如 48(H) 和 69(i) 经常一起出现，合并成 token Hi。
        2. 多字节字符合并： 对于中文“世”（E4 B8 96），这三个字节总是绑定出现的。
          • 如果“世”字常见，BPE 会学到规则：先合 E4+B8，再合 (E4B8)+96。
          • 最终，E4 B8 96 这三个字节会被合并成一个 ID，代表“世”。
        3. 生僻字处理： 如果遇到一个极罕见的汉字（例如由 E5 99 99 组成），且词表中没有合并它的记录。那么它就不会被合并成一个 token，而是被拆解为 3 个代表字节的 token。

        第四步：最终 Tokenization

        • 常见的词（如 “apple”, “中国”）会对应一个独立的 Token ID。
        • 不常见的词，会退化为几个 Token ID（子词或字节）。
        • 完全未知的乱码或二进制数据，会退化为单字节 Token ID。

        5. 关键技术细节：GPT-2 的 Trick

        直接对原始字节运行 BPE 有个小问题：有些字节是控制字符（不可打印），有些字节组合在处理时比较麻烦。

        GPT-2 的作者在实现 Byte-Level BPE 时使用了一个技巧：

        • 他们不直接使用 0-255 的原始数值。
        • 他们定义了一个可逆映射函数，将 256 个字节值映射为 256 个可打印的 Unicode 字符（避开控制字符和空格问题）。
        • 例如：字节 0 可能映射成字符 Ā，字节 1 映射成 ā…
        • 分词是在这些映射后的字符上进行的，解码时再反向映射回字节。
        • 注：这主要是为了兼容当时已有的 BPE 训练工具，逻辑本质上还是对字节操作。

        6. 优缺点分析

        优点：

        1. 彻底解决 OOV（未登录词）问题： 只要是计算机能存的数据，都能分词，绝对不会出现 <UNK>。
        2. 多语言通用性强： 不需要针对中文、英文、阿拉伯文分别设计分词器，一套词表走天下。
        3. 词表大小可控： 通常设定在 50k - 100k 左右，既能覆盖常用词，又能处理生僻词。
        4. 支持非文本数据： 理论上可以直接处理代码、Emoji 甚至二进制数据。

        缺点：

        1. 非拉丁语系的 Token 效率稍低：
          • 英文单词通常 1 个单词 = 1 个 Token。
          • 中文虽然很多常见字也是 1 个 Token（因为 3 个字节被合并了），但如果遇到稍微生僻点的词，可能一个字会被拆成 2-3 个 Token（字节碎片）。这导致在相同上下文窗口（Context Window）下，中文能包含的信息量有时不如专门为中文优化的分词器多。
        2. 字节碎片无效性： 如果模型生成的 Token 序列在中间截断了（例如生成了“世”字的前两个字节，缺了第三个），解码出来的字符串就是乱码（无效的 UTF-8）。需要额外的解码逻辑来忽略或修复这些错误。

        7. 举个直观的例子

        假设词表大小限制很小，模型遇到了字符串 “Earth地球”：

        1. Earth: 常见英文。

          • UTF-8字节: 45 61 72 74 68
          • BPE结果: 可能会合并成一个 Token [Earth]。

        2. 地: 常见中文。

          • UTF-8字节: E5 9C B0
          • BPE结果: 因为常见，这三个字节已经在词表中合并了，变成一个 Token [地]。

        3. 㙓 (生僻字): 假设这是个从未见过的字。

          • UTF-8字节: E土 土 土 (假设编码为 F0 A0 B0 C0，共4字节)
          • BPE结果: 词表中没有 [㙓] 这个整体。
          • 模型会将其拆分，也许拆成两个 Token: [F0A0] 和 [B0C0] (假设这两个字节对以前见过)，或者直接拆成 4 个单字节 Token。
          • 关键点： 虽然拆碎了，但信息没丢，模型依然有机会通过字节的组合去理解它的含义（或者至少能原样复读出来）。

        总结

        基于字节的 Tokenizer 是大模型时代的通用语。它通过将文本拆解为最底层的字节流，再利用 BPE 算法自底向上地重组出有意义的子词，成功实现了在有限词表下对人类所有语言和符号的无损编码。
